# Mocking GUI System Architecture & Logic Reference

This document provides an in-depth analysis of the overall structure, functional design, and core logic of the Mocking GUI project. All agents must refer to this reference as the highest priority when working on Mocking GUI core development and related tasks.

---

## 1. Mocking GUI Ecosystem Overview

Mocking GUI is a library that uses **MSW (Mock Service Worker) 2.x** as its engine and provides a **GUI panel** for user-friendly control.

- **Core Engine**: Intercepts browser-level network requests using MSW's `http` interceptor.
- **State Layer**: Manages the activation state, selected response cases (Variants), and delay times of handlers through a Zustand-based store.
- **Sync Layer**: Uses the browser's local storage and cookies to maintain state even across page refreshes and in server-side rendering (SSR) environments.

---

## 2. Core Components

### 2.1 MockingGUIBoundary (`packages/mocking-gui/src/components/MockingGUIBoundary.tsx`)

The top-level entry point of the Mocking GUI library.

- **Role**: Responsible for MSW worker initialization, initial state loading, and GUI panel rendering.
- **Initialization flow**: Renders `LoadingPage` while `isReady = false`.
  1. Receives the `MockingConfig` prop and performs 3-phase initialization via `useSetupMockingGUIWorker`.
  2. `InternalUI` — renders `MockingGUIPanel` in isolation inside `ShadowRootPortal`.

### 2.2 MockingGUI WorkerManager (`packages/mocking-gui/src/utils/browser/workerManager.ts`)

A module-level singleton (not a class instance) that manages the MSW `setupWorker` instance.

- **Module-level state**: Singleton guaranteed via `worker`, `startPromise`, and `isHandlerStarted` variables.
- **Prevents duplicate start**: Returns existing Promise when `startPromise` exists to prevent double initialization.
- **Dynamic Update**: Re-injects updated handlers via `convertToMswHandler()` → `worker.resetHandlers()` when `handlerConfigs` changes.

### 2.3 Handler Store (`packages/mocking-gui/src/store/useHandlerStore.ts`)

The core state store implemented with Zustand.

- **State structure**: Manages real-time settings for each handler in `Record<string, StoredHandlerVariants>` format.
- **Persistence**: Uses the `persist` middleware to preserve state even after the browser is closed.
- **Cookie synchronization**: Updates specific cookies on state changes so that server-side code can also read the information.

## 3. Request Handling Logic

### 3.1 Handler Adapter (`convertToMswHandler`)

The core adapter that converts Mocking GUI's high-level configuration (`HandlerState`) into MSW's `HttpHandler`.

| Mode (Type) | Behavior                                                               | Key Logic                                                             |
| :---------- | :--------------------------------------------------------------------- | :-------------------------------------------------------------------- |
| **MANUAL**  | Returns the selected case from user-defined static cases               | Searches `responseVariants` array and generates `HttpResponse.json()` |
| **AUTO**    | Dynamically generated by analyzing request parameters/body             | Returns the result of executing `responseVariantsFn` as the response  |
| **SWAGGER** | Uses automatically generated responses extracted from the Swagger spec | Response handling using `swaggerResponseVariants`                     |

### 3.2 Core Conversion Process

- **State check**: If not active (`active`), returns `passthrough()` to send the request to the real server.
- **Delay simulation**: Calls MSW's `delay()` function when `delayTime` is configured.
- **Response generation**: Packages body, status code, and headers using `HttpResponse.json()`.

## 4. Directory Structure Analysis

### 4.1 `packages/mocking-gui/src`

- **`store/`**: Handler state, scenarios, and Swagger source management (Zustand + persist).
- **`utils/browser/`**: MSW Worker singleton + `mocking_gui_sync` cookie synchronization.
- **`utils/handler/`**: Mocking GUI → MSW adapter (`convertToMswHandler`).
- **`utils/swagger/`**: OpenAPI loading/conversion/merging/export.
- **`utils/server/`**: SSR server creation + cookie → state restoration.
- **`types/`**: Core interfaces (`MockingConfig`, `HandlerConfigOption`, `RawBody`, `Scenario`, etc.).

### 4.2 `docs/`

- **`guide/`**: Conceptual introduction, installation guide.
- **`guide/usage/`**: Detailed guides per feature (API, GUI, Scenario, Swagger).

---

## 5. Engineering Standards and Constraints

1. **Strict Type System**: Use of `any` is prohibited; common interfaces under `types/` must always be extended when adding new handlers or features.
2. **HTTP Only**: Currently only the MSW `http` namespace is supported; GraphQL/WebSocket, etc., are managed as future extension tasks.
3. **Integrity First**: State changes must only be made through the store, and immutability must be maintained.
4. **Server-Side Sync**: If mocking state does not match on the server side, first inspect the cookie parsing logic in `packages/mocking-gui/src/utils/server/setup.ts`.
