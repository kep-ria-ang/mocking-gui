# Mocking Strategy: Domain-Driven Data Modeling

A domain-driven mock data design strategy based on the 4-Layer pattern.

## 1. Resource ID Chain Pattern

Defines rules for ID generation and referencing to manage relationships between complex resources.

- **Rule**: Sub-resources must be designed to include or reference the parent resource's ID.
- **Benefit**: Maintains data consistency across APIs and enables realistic scenario construction.

## 2. Dynamic Resolvers with Immer

Uses the `Immer` pattern inside `responseVariantsFn` to safely handle complex state changes.

- **Strategy**: Process and return the original data generated by factories based on request parameters (filters, pagination).

## 3. Atomic Data Isolation

Each domain (`user`, `product`, etc.) has its own `constants` and `factories` and does not directly depend on another domain's internal data. When integration is needed, it is done through shared `utils`.
